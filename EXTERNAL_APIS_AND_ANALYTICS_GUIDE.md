# External APIs And Analytics Guide

This app is close to a usable MVP, but it is not production-ready yet.

Right now, the product flow works, but several parts are still local or mocked:

- Content is hardcoded in `src/data/mockData.ts` and related data files.
- Preferences, favorites, enquiries, and language are stored locally in `src/store/AppContext.tsx` with `AsyncStorage`.
- The insights screen uses mock data in `app/(app)/insights.tsx`.
- Recommendations and chatbot behavior are local business logic, not connected to a live backend.

So the short answer is:

- Yes, the app can function as an MVP.
- No, "real analytics" alone is not enough for a real launch.
- You also need real data sources, a backend or proxy layer, monitoring, and privacy-safe configuration.

## 1. What you should connect first

The next production step should be split into 3 separate integrations:

### A. Content API

Use this for:

- destinations
- activities
- accommodations
- restaurants
- chauffeur profiles
- itineraries

This replaces hardcoded content files over time.

### B. Enquiry / CRM API

Use this for:

- sending enquiry form submissions to a real inbox or CRM
- storing leads in a database
- tracking enquiry status
- connecting to a concierge workflow

This is more important than analytics if the goal is real business usage.

### C. Analytics

Use this for:

- screen views
- favorites added
- enquiries submitted
- top destinations viewed
- recommendation usage
- chatbot usage

This helps you understand usage, but it should not be the only production integration.

## 2. Important architecture rule

Do **not** connect sensitive third-party services directly from the mobile app if they require secret keys.

Bad pattern:

- App -> Google Analytics Data API
- App -> CRM admin API
- App -> database with secret credentials

Good pattern:

- App -> your backend API
- Your backend API -> Google Analytics, CRM, CMS, database, email provider

Why:

- mobile apps are distributed to users
- anything inside the app bundle can be inspected
- Expo `EXPO_PUBLIC_*` values are not secret

## 3. Recommended architecture for this project

Use this shape:

```text
Expo app
  -> your API server / backend-for-frontend
  -> CMS / database / CRM / analytics providers
```

Recommended responsibilities:

- App:
  - UI
  - local cache
  - optimistic state
  - public configuration only
  - analytics event emission

- Backend:
  - authentication if needed later
  - secret keys
  - rate limiting
  - data transformation
  - GA4 Data API access
  - CRM integration
  - email sending

## 4. What "real analytics of Pur Life Maroc" can mean

There are two different analytics sources, and they should not be confused.

### Website analytics

If Pur Life Maroc already uses Google Analytics 4 on the website, you can pull website reporting from the Google Analytics Data API and show a curated summary in the app.

Example uses:

- top destination pages on the website
- top countries of visitors
- most viewed offers
- campaign performance

This is useful for the admin side or for an internal dashboard-like screen.

### App analytics

This is event tracking generated by the app itself.

Example events:

- `screen_view`
- `favorite_added`
- `destination_opened`
- `recommendations_generated`
- `enquiry_submitted`
- `chat_prompt_sent`

This is what tells you how people actually use the mobile app.

Website analytics and app analytics are related, but they are not the same dataset.

## 5. Environment variables in Expo

For this Expo project, public runtime config should use `EXPO_PUBLIC_*` variables.

Example `.env.local`:

```env
EXPO_PUBLIC_API_BASE_URL=https://api.purlife-maroc.com
EXPO_PUBLIC_APP_ENV=development
EXPO_PUBLIC_ANALYTICS_HOST=https://us.i.posthog.com
EXPO_PUBLIC_ANALYTICS_KEY=your_public_analytics_key
```

Rules:

- only put public values in `EXPO_PUBLIC_*`
- never put service-account JSON, admin tokens, CRM secrets, or database passwords in the app
- use a backend for anything sensitive

## 8. Minimal API client example

Create `src/services/api/client.ts`:

```ts
import { env } from '../../config/env';

export async function apiRequest<T>(
  path: string,
  init?: RequestInit
): Promise<T> {
  const response = await fetch(`${env.apiBaseUrl}${path}`, {
    headers: {
      'Content-Type': 'application/json',
      ...(init?.headers || {}),
    },
    ...init,
  });

  if (!response.ok) {
    throw new Error(`API request failed: ${response.status}`);
  }

  return response.json() as Promise<T>;
}
```

## 9. Example enquiry API integration

Today, `addEnquiry` in `src/store/AppContext.tsx` only stores locally.

For production, keep the local save if you want offline resilience, but also call the backend:

```ts
import { apiRequest } from './client';
import type { Enquiry } from '../../types';

export async function submitEnquiry(enquiry: Enquiry) {
  return apiRequest('/enquiries', {
    method: 'POST',
    body: JSON.stringify(enquiry),
  });
}
```

Then inside your submit flow:

1. save locally
2. call backend
3. show success or retry state

That gives you a real concierge pipeline instead of only a local record.

## 10. Example content API integration

Instead of importing all content from `src/data/mockData.ts`, move screens toward API-backed loaders.

Example:

```ts
import { apiRequest } from './client';
import type { Destination } from '../../types';

export async function fetchDestinations() {
  return apiRequest<Destination[]>('/content/destinations');
}
```

Then screen components can:

- fetch on mount
- store in state
- render loading / error / success states
- optionally cache results in `AsyncStorage`

## 11. Example analytics service wrapper

Even if you choose a provider later, create one analytics wrapper now.

Create `src/services/analytics/analytics.ts`:

```ts
type AnalyticsEvent =
  | 'screen_view'
  | 'favorite_added'
  | 'destination_opened'
  | 'recommendations_generated'
  | 'enquiry_submitted'
  | 'chat_prompt_sent';

export const analytics = {
  async track(event: AnalyticsEvent, properties?: Record<string, unknown>) {
    if (__DEV__) {
      console.log('[analytics]', event, properties || {});
    }

    // Replace this block with the SDK or HTTP call of the provider you choose.
  },

  async identify(userId: string, traits?: Record<string, unknown>) {
    if (__DEV__) {
      console.log('[analytics identify]', userId, traits || {});
    }
  },
};
```

Why this matters:

- screens stay independent from the analytics vendor
- changing provider later is easier
- analytics calls stay consistent across the app

## 13. Replacing the current mock insights screen

Today, `app/(app)/insights.tsx` uses:

```ts
const insights = useMemo(() => insightsUtils.generateMockInsights(), []);
```

For production, replace that with an API call:

```ts
const [insights, setInsights] = useState<InsightsData | null>(null);
const [isLoading, setIsLoading] = useState(true);

useEffect(() => {
  let isMounted = true;

  const loadInsights = async () => {
    try {
      const data = await apiRequest<InsightsData>('/analytics/overview');
      if (isMounted) {
        setInsights(data);
      }
    } finally {
      if (isMounted) {
        setIsLoading(false);
      }
    }
  };

  void loadInsights();

  return () => {
    isMounted = false;
  };
}, []);
```

Your backend can shape the response to match the existing `InsightsData` type, which keeps UI changes small.

## 14. If Pur Life Maroc already has Google Analytics 4

If the company already tracks the website in GA4, the clean approach is:

1. Create a backend endpoint such as `GET /analytics/overview`
2. Let the backend call the Google Analytics Data API
3. Transform the vendor response into your own stable JSON shape
4. Return only the fields the app actually needs

Do not call the GA Data API directly from the app because that setup usually relies on protected credentials and property access.

A backend response could look like this:

```json
{
  "topTravelStyle": { "name": "Bespoke Travel", "count": 122 },
  "topDestination": { "name": "Essaouira", "count": 341 },
  "topActivity": { "name": "Argan Oil Experience", "count": 87 },
  "topAccommodation": { "name": "Villa Quieta", "count": 64 },
  "topItinerary": { "name": "Atlantic Art de Vivre", "count": 31 },
  "topDriver": { "name": "Samira", "count": 22 },
  "trendingExperiences": [
    { "name": "Essaouira", "trend": 18 },
    { "name": "Marrakesh", "trend": 12 }
  ]
}
```

That matches your current UI model much better than exposing raw GA rows to the app.

## 15. Rollout order I recommend

Build in this order:

1. Real enquiry backend
2. Content API for destinations, activities, accommodation, and restaurants
3. Analytics wrapper and event tracking in the app
4. Real insights endpoint
5. Error monitoring
6. Auth only if the business workflow actually needs it

This order gives business value early.

## 16. Common mistakes to avoid

- putting secrets in `EXPO_PUBLIC_*`
- calling admin APIs directly from the app
- tying screens directly to raw third-party responses
- assuming website analytics automatically equals app analytics
- replacing all local state at once instead of migrating screen by screen
- shipping analytics without a privacy policy / consent decision

## 19. Final recommendation

For this project, the best real-world setup is:

- Expo app for the client
- one backend API owned by Pur Life Maroc
- backend connections to CMS / CRM / GA4 / email
- app-side analytics wrapper for product events
- gradual replacement of mock data screen by screen

That is the safest way to turn this MVP into a real product without hard-coding business secrets into the mobile app.
